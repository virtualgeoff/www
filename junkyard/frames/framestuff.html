Prevent scripting errors

 		  
Although HTML frames are a stable specification, frame behavior in the document object model is less well-defined. Particular problems arise with scripts that execute as the frameset page is loading. Fortunately, you can fix them all with a single operation.

The first problem is that the document events onLoad and onUnload are associated with the <BODY> element, which frameset pages normally don't have (and frame-capable browsers would ignore anyhow). The most recent browsers let you put these event handlers in the uppermost <FRAMESET> tag, but some of the various 3.0 browsers do not reliably support these handlers.

Many browsers also have problems with frameset page scripts that modify frame contents immediately. This is due to conflicts between loading the document specified in the <FRAME> tag and following scripted instructions to load or write something else. These browsers often execute the JavaScript command, and then proceed to load the initial src page anyhow. And some browsers don't recognize a given frame's existence until its initial src document has loaded. Until it has loaded, any script reference to the frame will produce a JavaScript error.

The solution to all of these problems is to make sure the frame contents are done loading before you mess with them. Therefore, it's a good idea to begin all your frames with HTML pages--very simple ones, if necessary--and have them report to the master frameset. For example, suppose you have a frameset page that needs to load all frames before executing the function goToIt(). Place the following JavaScript in the frameset document:

countDown=frames.length;
function soundOff() {
 countDown--;
 if (countDown==0) {
  goToIt();
  }
 }

Then put onLoad="parent.soundOff()" in the <BODY> of each framed page. As the framed pages load and execute soundOff(), countDown ticks off until all the frames are accounted for.
